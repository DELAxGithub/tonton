# Test-First ルール（Agent Harness）

非自明な実装タスクに着手する前に、以下を**必ず**順守する。
トリビアル（typo 修正・1 行変更・リネーム）は対象外。

## 1. テスト観点の明示（Red before Green）

コードを書く**前**に、以下を会話に 1 ブロックで提示する:

```
### テスト観点
1. <観点1: 正常系の振る舞い>
2. <観点2: 境界条件>
3. <観点3: エラー経路 / 失敗時の挙動>
(必要に応じて 4〜5 まで)

### どこに書くか
- <test ファイルパス>
```

3〜5 点、1 行ずつ。不要に増やさない。
ユーザが「プロトタイプ優先」「動くもの優先」と指示している場合は
観点 1 点でよい（ただし必ず 1 点は書く）。

## 2. テスト雛形を先に置く

既存テスト基盤がある場合:
- Node: `*.test.ts` / Vitest / Jest
- Python: `tests/test_*.py`
- Flutter: `test/*_test.dart`
- iOS: `*Tests.swift`

実装より先に **failing test**（`expect(…).toBe(…)` など）を書く。
テスト基盤が無いプロジェクトでは、そのことを明示してから実装に入る。

## 3. 実装 → 緑確認 → コミット

- 実装後、CI ではなくローカルで該当テストを実行して緑を確認する
- 緑になってから `git add` / `git commit`
- CI の `auto-fix.yml` は「人間がテストを書き忘れた時の最後の砦」であり、
  ガードレール本体ではない

## 4. LLM 生成物の自己検証

[`@delax/review-commons`](../../../packages/review-commons/) が利用可能なプロジェクトでは、
生成されたコードに対して `ReviewFinding` schema の JSONL を出力し、
`confidence` が `low` なら**ユーザに確認**を求める。

## 5. テストファイル不変条件（テスト詐欺対策）

実装フェーズで**テストファイルを書き換えない**。

AI エージェントは「テスト PASS」を目的化しがちで、テストそのものを書き換えて
通す挙動（assert 弱体化 / skip 追加 / ダミーデータ埋め / 期待値の差し替え）が出る。
これを防ぐために以下を守る：

- 実装フェーズに入ったら、テストファイル（`*.test.ts` / `*.spec.ts` /
  `tests/test_*.py` / `*_test.go` / `*_test.dart` / `*Tests.swift` / `*Test.kt` 等）は
  **read-only として扱う**
- テストの「観点追加」「ケース追加」「assert 強化」は許可。
  「弱体化 / skip / ダミー化 / 期待値の都合付け替え」は **禁止**
- 例外: テスト基盤の追加 / テストリファクタ自体が目的のときは、その旨を会話と
  コミットメッセージに 1 行で明示する

**レビュー観点（人間 + `/review` skill 共通）**:
今回の差分で **既存テストファイルが書き換わっていないか**。書き換わっている場合：
- 観点追加 / assert 強化 → OK
- skip 追加 / 期待値の差し替え / ケース削除 → **要正当化、原則 reject**

> 機械的検証（mtime / git log ベース）は `actions/checkout` の影響などで誤検出が多く、
> ノイズが効果を上回るため現時点では入れていない。将来 `@delax/review-commons` の
> `ReviewFinding` 経由で実装する候補。

## なぜこれを守るのか

- AI エージェントは「通ったテスト」を唯一の信頼できる reward signal とする
- テスト観点がないまま生成されたコードは、人間が読むまで品質不明
- 事後テスト（後から追加）は「実装に合わせたテスト」になり、ガードレールにならない

## 例外

- `docs/` のみの変更、`.md` 文書化、設定ファイルの単純な key 追加
- ユーザが明示的に「テストなしで良い」と言った場合（その旨を 1 行で記録）
