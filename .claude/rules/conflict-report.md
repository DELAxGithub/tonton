# Conflict Report ルール（Agent Harness）

実装中に **要件の衝突 / 前提の不一致 / 仕様の曖昧さ** に遭遇したとき、
**AI は推測で解決してはならない**。代わりに `Conflict Report` を出力して人間の介入を待つ。

「もっともらしい妥協案を捏造する」のは AI の最悪の失敗モードであり、
バグより発見が遅く、後工程で不可逆な負債になる。

## 1. 発動条件

以下のいずれかに該当したら、実装を**中断**して Conflict Report を出力する:

- 要件 A と要件 B が物理的に両立しない（例: 「即時応答」と「全件同期チェック」）
- 既存コードの前提と新仕様が矛盾する（例: スキーマ変更が既存マイグレーションを破壊）
- 渡された情報だけでは決定不能な分岐がある（例: エラー時にリトライするか中断するか）
- セキュリティ / プライバシー要件が機能要件と衝突する

トリビアルな曖昧さ（変数名、コメント文言）は対象外。
**「load-bearing な前提が崩れた時だけ」** 発動する。

## 2. 出力フォーマット

```markdown
## Conflict Report

### 衝突の種類
- requirement-vs-requirement / requirement-vs-existing / under-specified / security-vs-feature

### 何と何がぶつかっているか
- A: <要件 / 前提 / 既存実装の説明>
- B: <要件 / 前提 / 既存実装の説明>

### 私が勝手に決めない理由
- <この決定が load-bearing である理由を 1〜2 行>

### 想定される選択肢（推測ではなく列挙）
1. <選択肢 1> — トレードオフ: <得るもの / 失うもの>
2. <選択肢 2> — トレードオフ: <得るもの / 失うもの>
3. <選択肢 3>（ある場合）

### 必要な人間の判断
- <ユーザーに何を答えてほしいか、1 行で>
```

## 3. 紛争解決ポリシーの優先順位

プロジェクトに明示がない場合、デフォルトの優先順位:

```
セキュリティ > データ整合性 > ユーザー体験 > パフォーマンス > 開発速度
```

ただし**この順位を根拠に勝手に解決してはならない**。
あくまで人間が判断する際のヒントとして Conflict Report 内で参照する。
プロジェクト固有のポリシーがあれば `CLAUDE.md` に上書き定義する。

## 4. サーキットブレーカー

同一トピックで Conflict Report が **3 回連続** で発生したら、
個別の判断ではなく **設計そのものの見直し** を提案する。

- 「このタスクは仕様が未成熟です。要件定義に戻ることを推奨します」
- 該当箇所を `docs/conflicts/YYYY-MM-DD-<topic>.md` に保存して、
  後で `/adr` で正式な意思決定として記録する

## なぜこれを守るのか

- AI は「沈黙」より「もっともらしい嘘」を選ぶ傾向がある（Hallucination）
- 妥協案の捏造は、コードレビューでも発見しにくい（一見動くため）
- Conflict Report で**判断を可視化**することで、後の ADR・テストケース・
  バグ報告に再利用できる

## 例外

- ユーザーが明示的に「お任せで」と言った場合（その旨を 1 行で記録）
- プロトタイプ段階（feedback `prototype-first` 適用時）— 動くものを優先
