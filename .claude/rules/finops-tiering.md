# FinOps Tiering ルール（Agent Harness）

すべての変更に最大強度のレビューループを適用するとトークン消費で破綻する。
**機能の重要度に応じて規律の強度を使い分ける** のが 1 人〜小規模チーム開発における鉄則。

## Tier 定義

### Tier 1 (Core) — 最大強度
**対象**: 決済、認証 / 認可、個人情報の取り扱い、不可逆な破壊的操作、
データベースマイグレーション、外部公開 API、課金ロジック

**適用フロー**:
1. 実装前に `Conflict Report` を出すか確認（前提が load-bearing なら明示）
2. テスト先行（`.claude/rules/test-first.md` 厳守）
3. `/review` でセルフレビュー
4. `/moderate code <range>` で **独立 subagent による判定**
5. 人間（ユーザー）が最終承認 — Human-Oracle
6. 決定理由を `/adr` で `docs/adr/` に永続化

### Tier 2 (High) — 標準強度
**対象**: 新機能の主要ロジック、外部 API 連携、複雑な UI 状態管理、
ビジネスロジックのリファクタ、CI / デプロイ設定変更

**適用フロー**:
1. テスト観点を 1 ブロック明示（`test-first` 軽量版）
2. 実装
3. `/review` でセルフレビュー
4. ローカルテスト緑を確認してコミット

### Tier 3 (Normal) — 軽量
**対象**: 文言変更、CSS 微調整、既存画面の小修正、
コメント追加、依存パッケージのパッチ更新、devlog / README 更新

**適用フロー**:
1. 実装
2. diff を一読
3. コミット（`/review` 不要）

## 判定の優先順位

複数 Tier に該当しそうな変更は、**最も高い Tier** を採用する。

迷ったら Tier 2。Tier 3 と判断したら理由を 1 行コミットメッセージに含める。

## どの Tier かを宣言する

非自明な実装に入る前に、以下のフォーマットで宣言する:

```
### Tier 判定
- Tier: <1 / 2 / 3>
- 理由: <1 行>
- 適用フロー: <上記の番号付きステップを列挙>
```

これにより、後から「なぜこのレベルのレビューだったか」が追える。

## アンチパターン

- ❌ **すべて Tier 1 で回す** — トークン消費で破綻、レビュー疲れで形骸化
- ❌ **Tier 3 で済ませて事故** — 認証ロジックを「ちょっとした変更」と判断する等
- ❌ **Tier を後から下げる** — レビューが面倒になったから格下げするのは禁止
- ❌ **Tier 1 でテストなしを許容** — Tier 1 ではテスト先行が必須

## プロジェクト固有のオーバーライド

プロジェクトの `CLAUDE.md` で対象範囲を上書き可能:

```markdown
## Tier overrides
- src/payment/**     → Tier 1 強制
- src/components/**  → 通常 Tier 2、新規追加は Tier 3 可
- docs/**            → Tier 3 固定
```

## なぜこれを守るのか

- AI エージェントの呼び出しコストは「無料」ではない
- すべての PR に多重レビューをかけると人間も AI も疲弊し、
  本当にレビューすべき Tier 1 の集中力が落ちる
- Tier 宣言を残すことで、後の振り返り（事故った時、効果測定したい時）に
  「適切なレベルだったか」を判定できる

## 関連
- `.claude/rules/test-first.md` — Tier 1〜2 で必須
- `.claude/rules/conflict-report.md` — 全 Tier 共通
- `/review` — Tier 2 以上の標準ゲート
- `/moderate` — Tier 1 の独立判定ゲート
- `/adr` — Tier 1 の決定理由を永続化
